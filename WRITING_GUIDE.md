# 王炜博客写作规范

> 本文档定义了王炜个人博客的文章写作标准。任何 AI 或人类作者在阅读本文档后，应能独立产出风格、结构、质量一致的内容。

---

## 一、定位与读者

### 1.1 目标读者

企业管理层：CEO、CIO、CSO、部门总经理，以及有一定经验的安全项目经理。

### 1.2 读者特征

- 关心投入产出比，而非技术细节
- 需要结构化信息，快速抓住重点
- 希望获得可执行的建议，而非纯理论
- 对行业数据和案例有信任感
- 厌恶空洞说教和贩卖焦虑

### 1.3 核心定位

**站在企业治理与项目交付视角，拆解安全工作怎么建体系、怎么推落地、怎么做闭环。**

不是站在纯技术攻防视角，也不是站在纯理论咨询视角。

---

## 二、文风要求

### 2.1 基本原则

| 要求 | 说明 | 反例 |
|-----|------|------|
| 完整句式 | 优先使用连贯段落，减少碎片化 bullet points | ❌ "AI能做好三件事：框架搭建、背景整理、语言润色" |
| 结构化强 | 每篇文章有清晰的框架，管理层能 10 秒看懂逻辑 | 文章没有明确的问题-分析-解法结构 |
| 有数据/案例 | 不说空话，用行业数据或典型案例支撑 | "很多企业都这样做"（没有具体数据） |
| 有结论有建议 | 不止于描述问题，要给出行之有效的行动建议 | 只分析问题，不给解法 |
| 语言克制 | 不用夸张修辞，不贩卖焦虑 | "不这样做企业就会完蛋" |
| 专业但不晦涩 | 避免过多技术术语，必要时翻译成管理语言 | 大量 CVE 编号、工具参数 |

### 2.2 句式规范

**优先使用完整段落叙述，将碎片信息整合为连贯论述。**

**示例转换：**

**❌ 碎片化写法（禁止）：**

> AI 可以做好三件事：
> - 框架搭建
> - 背景整理
> - 语言润色

**✅ 完整句式写法（推荐）：**

> 从我们的实践经验来看，AI 目前在安全方案编写中最能发挥价值的环节主要集中在三个方面。首先是框架搭建，AI 可以在几十秒内生成一份结构完整的方案大纲，将原本需要数小时构思的框架设计压缩到几分钟。其次是背景整理，包括法规要求、行业案例、技术标准等信息的快速收集与结构化呈现，这部分工作传统上需要大量人工检索和筛选。第三是语言润色，将技术术语翻译为管理层易于理解的管理语言，显著提升方案的可读性和说服力。

**列表的使用边界：**

- 对比信息（如两种模式的差异）可以使用表格
- 检查清单类内容可以使用列表
- 其他场景优先使用段落叙述

## 三、文章结构模板

### 3.1 标准结构（每篇文章必须遵循）

```
1. 引言（约占 5%）
   - 用一句话点明核心观点
   - 用数据或现象引入话题
   - 说明本文要解决什么问题

2. 核心数据卡片（约占 3%）
   - 3 个关键数据，用大号数字展示
   - 数据可以是行业统计、调研结果或案例数据

3. 问题阐述（约占 20%）
   - 描述现象：读者能感同身受的具体场景
   - 分析根因：为什么这个问题会发生
   - 避免只描述不分析

4. 核心内容展开（约占 50%）
   - 每个子主题配一个视觉元素（图/表/案例框）
   - 先给框架，再给细节
   - 理论与实践结合

5. 案例复盘（约占 15%）
   - 真实或基于通用场景的案例
   - 包含背景、难点、解法、效果四要素
   - 用 case-study 组件包裹

6. 给管理层的建议（约占 5%）
   - 3-5 条可立即执行的行动建议
   - 用 tip-box 组件包裹

7. 结语 + 检查清单（约占 2%）
   - 金句收束全文
   - 5-6 条自检问题，用 checklist 组件
```

### 3.2 段落长度控制

- 每段不超过 150 字
- 段与段之间要有逻辑衔接
- 长段落拆分为两段，中间加过渡句

## 四、视觉元素规范

### 4.1 必须使用场景

| 位置 | 视觉元素类型 | 用途 |
|-----|------------|------|
| 引言后 | data-cards（3个数据卡片） | 用数据抓住注意力 |
| 每个主要章节 | 至少1个图表或表格 | 降低阅读门槛 |
| 核心观点 | insight-box（洞察框） | 金句高亮 |
| 风险提示 | warning-box（警示框） | 常见误区 |
| 行动建议 | tip-box（提示框） | 可操作建议 |
| 案例 | case-study（案例框） | 真实场景 |
| 文末 | checklist（检查清单） | 自检问题 |

### 4.2 图表制作规范

**Mermaid 图表：**
- 使用 architecture-beta、flowchart、sequenceDiagram
- 配色使用品牌色：深海蓝 `#1e3a5f`、琥珀橙 `#e67e22`、深绿 `#27ae60`、暗红 `#c0392b`
- 复杂流程图使用子图分组，避免过于密集

**数据图表（Python matplotlib）：**
- 配色：深海蓝主色、琥珀橙强调、深绿正向、暗红负向
- 必须添加数值标签
- 去除顶部和右侧边框
- 背景纯白，网格线虚线淡化

**图片尺寸：**
- 宽度控制在 800px 以内
- 高度不超过 600px
- 优先保留整图的完整结构感，避免为了适配网页而轻易拆图
- 如果图表信息密度较高，应优先通过点击放大、滚轮缩放和更高清原图来解决，而不是默认拆分
- 对于左右并列、强对比关系的流程图或机制图，可以合理拆分为两张子图，以减少网页横向压缩感，例如“现状 vs 优化后”“职责分散 vs 统一归口”这类图
- 只有在整体图经过放大浏览后仍然明显难读时，才考虑补充拆分版作为辅助图，而不是直接替代整图
- 正式图注优先使用 `图X：……` 的报告式写法，并放在图片上方
- 所有文章图片必须支持点击放大浏览，并适配深色浏览背景与缩放查看

## 五、配色方案

```
主色：#1e3a5f（深海蓝）—— 标题、架构图主色调
强调：#e67e22（琥珀橙）—— 核心洞察、关键数据
辅助：#5a7a8a（青灰）—— 次要文字、边框
成功：#27ae60（深绿）—— 最佳实践、正向案例
警示：#c0392b（暗红）—— 风险提示、常见误区
背景：#faf9f7（暖白）—— 页面底色
卡片：#ffffff（纯白）—— 内容卡片
边框：#e0ddd8（暖灰）—— 分割线、边框
```

## 六、图表生成流程

### 6.1 Mermaid 图表

1. 编写 `.mmd` 文件定义图表逻辑
2. 使用 `mmdc` 命令行工具生成 SVG：
   ```bash
   mmdc -i input.mmd -o output.svg -b transparent -w 800
   ```
3. 将 SVG 放入 `public/images/posts/postX/` 目录
4. 在文章中用 Markdown 图片语法引用：
   ```markdown
   ![描述](/images/posts/postX/filename.svg)
   ```

### 6.2 Python 数据图表

1. 编写 Python 脚本使用 matplotlib 生成
2. 保存为 PNG，分辨率 150dpi
3. 放入 `public/images/posts/postX/` 目录
4. 在文章中引用

### 6.3 图片交互规范

- 正文中所有图表必须放在 `.figure` 容器内，以启用点击放大功能
- 如果使用对子图分拆展示，则使用 `.figure-split + .figure-card + .figure-subcaption` 组合，并且只用于强对比关系的图
- 放大弹窗默认提供 `放大 / 缩小 / 还原` 控件，并支持鼠标滚轮缩放
- 图注需要使用正式报告语气，推荐格式：`图 3：漏洞全生命周期管理流程`
- 如果一张图需要读者停留超过 3 秒才能看清结构，优先提升原图清晰度与缩放体验，拆分只作为最后手段

## 七、文章元数据

每篇文章顶部必须包含 frontmatter：

```yaml
---
title: 文章标题
date: YYYY-MM-DD
category: 栏目名称
author: 王炜
description: 文章摘要（50字以内）
---
```

## 八、质量检查清单

文章发布前必须检查：

- [ ] 引言是否用一句话点明核心观点
- [ ] 是否包含 3 个数据卡片
- [ ] 每个主要章节是否配有图表或表格
- [ ] 是否有至少 1 个实战案例
- [ ] 是否有给管理层的行动建议
- [ ] 文末是否有检查清单
- [ ] 段落是否使用完整句式（非碎片化列表）
- [ ] 是否有核心洞察框高亮金句
- [ ] 数据是否标注来源或基于合理推演
- [ ] 文章长度是否在 5000-6500 字之间
