# 王炜博客写作规范

> 本文档定义了王炜个人博客的文章写作标准。任何 AI 或人类作者在阅读本文档后，应能独立产出风格、结构、质量一致的内容。

---

## 一、定位与读者

### 1.1 目标读者

企业管理层：CEO、CIO、CSO、部门总经理，以及有一定经验的安全项目经理。

### 1.2 读者特征

- 关心投入产出比，而非技术细节
- 需要结构化信息，快速抓住重点
- 希望获得可执行的建议，而非纯理论
- 对行业数据和案例有信任感
- 厌恶空洞说教和贩卖焦虑

### 1.3 核心定位

**站在企业治理与项目交付视角，拆解安全工作怎么建体系、怎么推落地、怎么做闭环。**

不是站在纯技术攻防视角，也不是站在纯理论咨询视角。

---

## 二、文风要求

### 2.1 基本原则

| 要求 | 说明 | 反例 |
|-----|------|------|
| 完整句式 | 优先使用连贯段落，减少碎片化 bullet points | ❌ "AI能做好三件事：框架搭建、背景整理、语言润色" |
| 结构化强 | 每篇文章有清晰的框架，管理层能 10 秒看懂逻辑 | 文章没有明确的问题-分析-解法结构 |
| 有数据/案例 | 不说空话，用行业数据或典型案例支撑 | "很多企业都这样做"（没有具体数据） |
| 有结论有建议 | 不止于描述问题，要给出行之有效的行动建议 | 只分析问题，不给解法 |
| 语言克制 | 不用夸张修辞，不贩卖焦虑 | "不这样做企业就会完蛋" |
| 专业但不晦涩 | 避免过多技术术语，必要时翻译成管理语言 | 大量 CVE 编号、工具参数 |

### 2.2 句式规范

**优先使用完整段落叙述，将碎片信息整合为连贯论述。**

**示例转换：**

**❌ 碎片化写法（禁止）：**

> AI 可以做好三件事：
> - 框架搭建
> - 背景整理
> - 语言润色

**✅ 完整句式写法（推荐）：**

> 从我们的实践经验来看，AI 目前在安全方案编写中最能发挥价值的环节主要集中在三个方面。首先是框架搭建，AI 可以在几十秒内生成一份结构完整的方案大纲，将原本需要数小时构思的框架设计压缩到几分钟。其次是背景整理，包括法规要求、行业案例、技术标准等信息的快速收集与结构化呈现，这部分工作传统上需要大量人工检索和筛选。第三是语言润色，将技术术语翻译为管理层易于理解的管理语言，显著提升方案的可读性和说服力。

**列表的使用边界：**

- 对比信息（如两种模式的差异）可以使用表格
- 检查清单类内容可以使用列表
- 其他场景优先使用段落叙述

### 2.3 开头写法规范

**开头必须避免模板化、空泛化、AI腔。**

管理层最容易对“假内容”失去耐心，而最容易暴露 AI 味的地方就是开头。如果一篇文章上来就是“在数字化转型背景下，网络安全越来越重要”“随着信息技术的快速发展，企业面临前所未有的挑战”这类套话，读者基本会立刻失去兴趣。

**开头禁止出现的写法：**

- “随着……快速发展”
- “在当今数字化时代”
- “网络安全已经成为企业必须面对的重要问题”
- “近年来，越来越多的企业开始重视……”
- 纯概念化、无场景、无冲突、无细节的空泛导入

**推荐的开头结构：**

1. **具体场景切入**：先写一个管理层熟悉的场景、决策瞬间或组织困境。
2. **矛盾或反差出现**：指出“投入很多但效果不好”“流程完整但闭环失败”“通过检查但能力没提升”这类张力。
3. **再给观点**：用一句判断性的话点出本文核心问题。

**推荐开头类型：**

- **场景式开头**：从会议、事故、汇报、整改现场切入。
- **反差式开头**：先写投入/表面成绩，再写真实问题。
- **问题式开头**：直接抛出管理层最关心的尖锐问题。
- **案例式开头**：用一个高度浓缩的真实场景开场。

**好的开头应具备的元素：**

- 有具体对象：谁在做决策、谁在承担压力
- 有明确场景：会议、项目、整改、验收、审计、汇报
- 有真实冲突：时间不够、责任不清、投入失衡、结果不达预期
- 有管理视角：强调组织、机制、风险、资源，而不是工具参数

**开头长度建议：**

- 第一段：40-90 字，快速给出场景或冲突
- 第二段：80-150 字，展开问题和反差
- 第三段：40-100 字，点出本文核心观点

**示例：差的开头（禁止）**

> 随着数字化转型不断推进，网络安全问题越来越受到企业重视。近年来，越来越多的企业开始加强安全建设，希望通过制度、流程和技术手段提升整体防护能力。

这类开头的问题是：没有人物、没有场景、没有冲突、没有观点，任何行业的任何文章都可以套用，读者无法感受到作者真的理解企业现场。

**示例：好的开头（推荐）**

> 安全预算刚批下来，设备也已经采购完成，但三个月后，管理层在项目复盘会上问的第一个问题依然是：“为什么高危漏洞还没有闭环？”问题并不在于企业没有投入，而在于安全工作从一开始就被当成了一次性建设项目，而不是一套持续运转的治理机制。很多企业的安全建设之所以看起来热闹、实际效果却一般，往往不是执行层不努力，而是顶层设计从第一步就偏了方向。

这个开头的特点是：有会议场景、有管理层问题、有投入与结果的反差，也有明确观点。

### 2.4 开头模块轮换机制

**后续文章不得长期使用同一种开头模块。**

如果连续几篇文章都采用“引言 + 三个数据卡片 + 进入正文”的同一模式，读者很快就会形成审美疲劳，并且会明显感觉到内容生产像模板流水线。为了避免这种“统一得过头”的问题，开头必须建立轮换机制。

**可选开头模块池：**

1. **场景引入型**
   - 先写会议、汇报、事故、验收、审计等具体场景
   - 再给出冲突和判断
   - 适合治理、项目、合规类文章

2. **反差对照型**
   - 先写“投入很多 / 表面通过 / 工具齐全”
   - 再写“结果一般 / 闭环失败 / 风险仍在”
   - 适合治理、漏洞、运营类文章

3. **单一关键数字型**
   - 只突出 1 个最有冲击力的数据，不用三卡片
   - 配一个短解释段落
   - 适合观点型、判断型文章

4. **双栏对照型**
   - 开头直接用一张小型对照表
   - 左边写常见做法，右边写真实后果
   - 适合“误区拆解”“两种模式对比”类文章

5. **案例切入型**
   - 先讲一个浓缩案例或复盘瞬间
   - 再抽象出本文主题
   - 适合漏洞治理、项目复盘、AI实践类文章

6. **问题追问型**
   - 用 2-3 个管理层会真正问的问题开头
   - 例如：“为什么预算批了，漏洞还没闭环？”
   - 适合偏管理和决策视角的文章

7. **判断先行型**
   - 第一段直接给结论
   - 第二段再解释为什么这样判断
   - 适合观点鲜明、争议较强的话题

**轮换要求：**

- 不允许连续 2 篇文章使用完全同一种开头结构
- 不允许连续 3 篇文章都使用“三个数据卡片”作为第一视觉模块
- 同一栏目内的相邻文章，开头节奏和视觉形式应尽量错开
- 如果上一篇用了“场景 + 数据卡片”，下一篇优先改成“案例 + 对照表”或“问题追问 + 单一关键数字”

## 三、文章结构模板

### 3.1 标准结构（每篇文章必须遵循）

```
1. 引言（约占 5%-8%）
   - 先用具体场景、反差或问题切入，而不是空泛背景句
   - 第一屏就让读者看到“为什么这篇文章值得继续读”
   - 用一句判断性的话点明核心观点
   - 再说明本文要解决什么问题

2. 核心数据卡片（约占 3%）
   - 3 个关键数据，用大号数字展示
   - 数据可以是行业统计、调研结果或案例数据

3. 问题阐述（约占 20%）
   - 描述现象：读者能感同身受的具体场景
   - 分析根因：为什么这个问题会发生
   - 避免只描述不分析

4. 核心内容展开（约占 50%）
   - 每个子主题配一个视觉元素（图/表/案例框）
   - 先给框架，再给细节
   - 理论与实践结合

5. 案例复盘（约占 15%）
   - 真实或基于通用场景的案例
   - 包含背景、难点、解法、效果四要素
   - 用 case-study 组件包裹

6. 行动建议（约占 5%）
   - 3-5 条可立即执行的行动建议
   - 用 tip-box 组件包裹

7. 结语 + 检查清单（约占 2%）
   - 金句收束全文
   - 5-6 条自检问题，用 checklist 组件
```

### 3.2 段落长度控制

- 每段不超过 150 字
- 段与段之间要有逻辑衔接
- 长段落拆分为两段，中间加过渡句
- 开头前三段必须有明显节奏变化，避免三段都写成同样的陈述句堆叠

### 3.3 文章范式库

**后续文章不应全部套用同一个“标准结构”。标准结构是底盘，不是外观。**

在不破坏整体质量的前提下，文章应当根据主题选择不同的表达范式。下面是可轮换使用的范式池。

#### 范式 A：管理诊断型

适合：治理体系、顶层设计、组织架构、指标设计。

结构：
- 真实管理场景
- 症状描述
- 根因拆解
- 框架图/对照表
- 管理建议
- 检查清单

特点：像一份短管理诊断报告，强调“问题在哪”。

#### 范式 B：案例复盘型

适合：漏洞闭环、项目推进、合规整改、跨部门协同。

结构：
- 案例开场
- 背景与挑战
- 过程拆解
- 难点与转折
- 结果与教训
- 可复制方法

特点：像一篇复盘文，强调“事情是怎么被推进下去的”。

#### 范式 C：决策备忘录型

适合：写给管理层的 AI、预算、组织投入、能力建设类文章。

结构：
- 先给判断
- 决策背景
- 支持判断的三到五个论据
- 风险提醒
- 建议动作

特点：像给管理层写的一页纸 memo，强调“应该怎么决策”。

### 3.4 内部模板库使用原则

模板库是**内部工作资产**，不是对外公开产品。它的作用是帮助写作者在动笔前快速选定结构、模块和表达节奏，而不是让读者一眼看出“这篇文章又套了哪个模板”。

因此，后续所有文章应遵循以下原则：

1. **写作前先选范式**
   - 在动笔前，从内部模板库中选择一个最适合当前主题的范式或模块组合。
   - 选择依据是主题特征、读者场景、冲突类型和视觉表达需求，而不是“哪个模板最顺手”。

2. **写作时消化模板，不暴露模板**
   - 模板只作为骨架使用，正文必须根据主题重新组织语言和节奏。
   - 不允许把模板标题、模板字段、模板语气原样搬到对外文章中。
   - 读者应该感受到文章“结构清楚”，而不是感受到“这是某个固定模版的复制品”。

3. **相邻文章必须错开感觉**
   - 即便都来自同一内部模板库，连续两篇文章在开头、首个视觉模块、正文推进方式上也必须有明显差异。
   - 要做到“内部有方法、外部不露痕迹”。

4. **模板是下限保障，不是上限限制**
   - 如果主题需要更适合的表达方式，可以在既有模板基础上混搭、重组或局部改写。
   - 目的始终是提升文章质量，而不是机械复用。

#### 范式 D：误区拆解型

适合：常见错误做法、认知偏差、表面建设等主题。

结构：
- 错误现象
- 为什么看起来合理
- 为什么实际上有问题
- 正确做法
- 一张强对比图或表

特点：强调“为什么很多人都做错了”。

#### 范式 E：路线图型

适合：建设方案、实施步骤、能力成熟度提升、年度推进路径。

结构：
- 起点现状
- 目标状态
- 分阶段路线图
- 每阶段关键动作
- 风险与保障机制

特点：像实施路线说明书，强调“接下来怎么做”。

#### 范式 F：问答拆解型

适合：管理层高频提问、争议性问题、AI是否可替代等主题。

结构：
- 抛出 2-4 个尖锐问题
- 逐个回答
- 用案例或图表补强
- 给最终判断

特点：读起来节奏快，适合阅读门槛较低但观点要求明确的文章。

**范式使用要求：**

- 不允许连续多篇文章都使用同一个范式
- 同一栏目中，优先交替使用“诊断型 / 案例型 / 决策型”
- 如果一篇文章已经用了大量框架图，下一篇应减少框架图、增加案例或对照结构

## 四、视觉元素规范

### 4.1 必须使用场景

| 位置 | 视觉元素类型 | 用途 |
|-----|------------|------|
| 引言后 | data-cards（3个数据卡片） | 用数据抓住注意力 |
| 每个主要章节 | 至少1个图表或表格 | 降低阅读门槛 |
| 核心观点 | insight-box（洞察框） | 金句高亮 |
| 风险提示 | warning-box（警示框） | 常见误区 |
| 行动建议 | tip-box（提示框） | 可操作建议 |
| 案例 | case-study（案例框） | 真实场景 |
| 文末 | checklist（检查清单） | 自检问题 |

### 4.1.1 开头视觉模块池

**开头视觉模块也必须轮换，不允许所有文章都使用三卡片。**

可选模块包括：

1. `data-cards`：三张数据卡片，适合数据感强的文章
2. `single-stat`：一张大数字主卡，适合单一关键判断
3. `comparison-table`：小型对照表，适合误区或模式对比
4. `case-study`：压缩案例卡，适合案例驱动型文章
5. `insight-box`：先亮观点，再展开正文，适合决策型文章
6. `timeline`：短时间线，适合路线图或复盘类文章

**建议：**

- 三张数据卡片不是默认项，只是备选项
- 如果主题本身缺少高质量数据，不要为了“看起来专业”硬塞三个框
- 开头视觉模块应服务内容，而不是反过来让内容去适配模板

### 4.2 图表制作规范

**Mermaid 图表：**
- 使用 architecture-beta、flowchart、sequenceDiagram
- 配色使用品牌色：深海蓝 `#1e3a5f`、琥珀橙 `#e67e22`、深绿 `#27ae60`、暗红 `#c0392b`
- 复杂流程图使用子图分组，避免过于密集

**数据图表（Python matplotlib）：**
- 配色：深海蓝主色、琥珀橙强调、深绿正向、暗红负向
- 必须添加数值标签
- 去除顶部和右侧边框
- 背景纯白，网格线虚线淡化

**图片尺寸：**
- 宽度控制在 800px 以内
- 高度不超过 600px
- 优先保留整图的完整结构感，避免为了适配网页而轻易拆图
- 如果图表信息密度较高，应优先通过点击放大、滚轮缩放和更高清原图来解决，而不是默认拆分
- 对于左右并列、强对比关系的流程图或机制图，可以合理拆分为两张子图，以减少网页横向压缩感，例如“现状 vs 优化后”“职责分散 vs 统一归口”这类图
- 只有在整体图经过放大浏览后仍然明显难读时，才考虑补充拆分版作为辅助图，而不是直接替代整图
- 正式图注优先使用 `图X：……` 的报告式写法，并放在图片上方
- 所有文章图片必须支持点击放大浏览，并适配深色浏览背景与缩放查看

### 4.3 组件使用语法

所有视觉组件通过 HTML `<div class="xxx">` 在 Markdown 中使用：

```markdown
<div class="insight-box">
  <p>核心洞察内容</p>
</div>

<div class="warning-box">
  <ul>
    <li>风险点1</li>
    <li>风险点2</li>
  </ul>
</div>

<div class="tip-box">
  <ul>
    <li>建议1</li>
    <li>建议2</li>
  </ul>
</div>

<div class="case-study">
  <h4>案例标题</h4>
  <p>案例内容...</p>
  <div class="case-result">
    <strong>效果：</strong>量化结果
  </div>
</div>

<div class="data-cards">
  <div class="data-card">
    <div class="number">67%</div>
    <div class="label">说明文字</div>
  </div>
</div>

<div class="single-stat">
  <div class="single-stat-number">35天</div>
  <div class="single-stat-label">平均漏洞修复时长</div>
</div>

<div class="checklist">
  <ul>
    <li>自检问题1</li>
    <li>自检问题2</li>
  </ul>
```

## 六、图表生成流程

### 6.1 Mermaid 图表

1. 编写 `.mmd` 文件定义图表逻辑
2. 使用 `mmdc` 命令行工具生成 SVG：
   ```bash
   mmdc -i input.mmd -o output.svg -b transparent -w 800
   ```
3. 将 SVG 放入 `public/images/posts/postX/` 目录
4. 在文章中用 Markdown 图片语法引用：
   ```markdown
   ![描述](/images/posts/postX/filename.svg)
   ```

### 6.2 Python 数据图表

1. 编写 Python 脚本使用 matplotlib 生成
2. 保存为 PNG，分辨率 150dpi
3. 放入 `public/images/posts/postX/` 目录
4. 在文章中引用

### 6.3 图片交互规范

- 正文中所有图表必须放在 `.figure` 容器内，以启用点击放大功能
- 如果使用对子图分拆展示，则使用 `.figure-split + .figure-card + .figure-subcaption` 组合，并且只用于强对比关系的图
- 放大弹窗默认提供 `放大 / 缩小 / 还原` 控件，并支持鼠标滚轮缩放
- 图注需要使用正式报告语气，推荐格式：`图 3：漏洞全生命周期管理流程`
- 如果一张图需要读者停留超过 3 秒才能看清结构，优先提升原图清晰度与缩放体验，拆分只作为最后手段

### 6.4 站内链接规范

**站内链接必须统一写法，禁止同一项目中同时混用多套规则。**

本项目当前采用 VitePress + GitHub Pages，并设置了 `base: /wangwei/`。这意味着链接写法必须考虑两类场景：Markdown 原生链接与 HTML 自定义锚点。

**规则如下：**

1. **VitePress 配置项中的链接**（如 `nav`、`sidebar`、首页 frontmatter 中的 `link`）
   - 统一使用根路径写法：`/about`、`/posts/governance-top5`

2. **Markdown 正文中的站内链接**
   - 统一使用根路径写法：`[标题](/posts/governance-top5)`、`[返回首页](/)`

3. **Markdown 中手写的 HTML `<a href="...">` 链接**
   - 统一使用相对路径写法：`./posts/governance-top5`、`./project`
   - 原因：原生 HTML 不会像 Markdown 链接那样自动适配 `base`，直接写死 `/wangwei/...` 不利于后续迁移，直接写 `/posts/...` 在 GitHub Pages 子路径下又容易出错

4. **禁止写死仓库名路径**
   - 禁止：`/wangwei/posts/xxx`
   - 原因：仓库名或部署路径一旦变化，这类链接会全部失效

**推荐示例：**

```markdown
<!-- Markdown 链接 -->
[查看文章](/posts/governance-top5)
[返回首页](/)

<!-- HTML 链接 -->
<a href="./posts/governance-top5" class="post-title">查看文章</a>
<a href="./project" class="post-title">查看栏目</a>
```

**检查要求：**

- 首页、文章总览页、栏目页、文章页必须遵循同一套链接规范
- 发布前搜索全站，确保没有残留 `/wangwei/` 这类写死路径
